03
1 / Agent

什么是 Agent

Agent 不是“更会聊天的模型”，而是围绕目标持续行动的系统。用户给它的是意图和结果，它要自己理解上下文、拆解步骤、调工具、交付成果。

  • 传统问答更像一次性输出；Agent 更像连续执行。
  • 它会根据任务动态判断下一步，而不是等用户逐条下命令。
  • 它依赖工具与反馈回路，不只是一个 prompt 包装层。
  • 真正的产品价值，在于缩短“意图到结果”的距离。
核心视角
Agent 是一种执行界面：用户说目标，系统组织动作，并对结果负责。

所以在产品设计上，重点不是回答是否流畅，而是目标、状态、动作、反馈和验收有没有被设计完整。

04
1 / Agent

从问答模型到执行系统

Understand

先理解用户是谁、目标是什么、约束和成功标准是什么。

Break Down

把复杂任务拆成若干步骤，决定优先级和可执行路径。

Act

读取文件、调用工具、运行脚本、产出中间结果。

Verify

检查结果是否可用，再决定结束、重试还是继续迭代。

对 PM 来说，真正要设计的是一套“可执行 + 可反馈 + 可收敛”的任务系统，而不是一句完美 prompt。

05
2 / Skill

Skill 的作用

Skill 可以理解成 Agent 的“标准作业指导书”。它提前规定何时触发、解决什么问题、先做什么、怎么验证、最后如何输出。

  • 把领域知识、执行顺序、约束条件和输出格式沉淀下来。
  • 减少随机发挥，让系统在特定场景更稳、更准。
  • 适合高频、边界清晰、可复用的任务类型。
  • 对产品来说，skill 是最容易规模化沉淀经验的能力单元。
skill = reusable expertise 
# 一个典型 skill 会提前说清这几件事
trigger: 什么意图会命中
scope: 解决哪类问题
workflow: 先看什么 / 再做什么 / 如何验收
guardrail: 边界、风险、不能做什么
output: 最后要以什么形式交付

2 / Skill

大部分情况下 Skill 就够用了

高频

问题重复度高，比如读文档、生成 slide、结构化总结、批量分析、常规改写。

窄域

输入输出相对稳定，边界清晰，不需要跨很多系统、角色或超长时间协同。

ROI

只要加上模板、规则和工具约束，就能快速把成功率和交付感拉起来。

  • 先做 20 个高频 skill，往往比一开始追求“万能 Agent”更快看到业务价值。
  • 对产品落地来说，先把单点能力做稳，比分散投入在全能幻想里更现实。
  • 只有任务开始跨系统、跨步骤、跨角色，workflow 的必要性才会明显上升。
08
3 / CLI

CLI 的作用

如果 Agent 只能生成建议，它的价值仍停留在辅助层。CLI 给了它一双手：读文件、改文件、跑脚本、执行命令、访问本地环境，把“我建议你这样做”变成“我已经替你做完这一步”。

  • CLI 连接的是执行环境，而不是仅仅提供一个工具列表。
  • 对内部效率场景尤其关键，因为大量资产本来就躺在代码仓库和脚本系统里。
  • 有了 CLI，Agent 才真正具备“操作世界”的能力。
一句话
没有 CLI，AI 停在建议层；有了 CLI，AI 才真正进入交付层。

所以 CLI 不是补充能力，而是让 Agent 从“会说”走向“会做”的关键桥梁。

4 / Workflow

Workflow 自动化

当任务由多个步骤组成时，单个 skill 已不足以覆盖全流程。Workflow 的核心不是“自动化”这三个字，而是编排：判断、调用、顺序、回写、验证、交接如何被组织起来。

  • 它把一个能力点，变成一条可重复执行的工作流。
  • 它需要处理状态流转、失败重试、人工介入点与上下游依赖。
  • 它更像业务流程设计，而不是单个能力设计。
编排视角
Workflow = 把多个 skill 和工具组织成一条可追踪、可治理、可复跑的业务链路。

自动化只是表象，真正重要的是流程被明确设计过，谁做什么、何时停、何时回传，都有规则。

4 / Workflow

Workflow 链路长什么样

1
输入收集读 PRD、看文档、拉上下文，先把任务补齐，而不是匆忙开跑。
2
能力路由命中合适的 skill 或工具，决定先后顺序，产出中间结果。
3
自动验证跑检查、看错误、评估结果是否达标，不把“验收”留到最后一刻。
4
人工接管风险高、结果冲突或信息不足时，把决策权交还给人。
13
5 / Stability

稳定性设计

Demo 能跑通，不代表产品可用。Agent 的真正难点，不是偶尔做对，而是大部分时候都做得足够对。稳定性设计，本质上是在管理不确定性。

  • 输入不确定：用户表达常常含糊、缺信息、甚至自相矛盾。
  • 过程不确定：模型有随机性，工具会失败，环境会变化。
  • 标准不确定：很多结果不是简单对错，而是主观阈值。
关键认知
稳定性不是优化项，而是 Agent 产品能不能被信任的前提。

目标不是消灭不确定性，而是把它收敛到产品可接受、可解释、可恢复的范围内。

14
5 / Stability

稳定性的常见抓手

1
输入约束减少任务歧义，限制问题空间，让系统先补足关键信息。
2
执行约束限定工具权限、调用顺序和可操作范围，避免失控动作。
3
结果校验对关键输出进行检查、重试、确认，而不是只看最终文本。
4
失败兜底不能成功时，也要给出可继续处理的状态、降级方案或人工接入点。
stability mindset 
# 一套简单但实用的 guardrail 逻辑
if input_missing:
  ask_or_infer_defaults()
elif high_risk_action:
  require_confirmation()
else:
  execute_safe_steps()

verify_each_step()
retry_or_fallback()
return_clear_status()

Takeaway

总结：一张图回收全场

Agent

目标驱动的执行系统，不只是一个更强的回答器。

Skill

把高频任务经验沉淀成可复用能力，是最现实的第一步。

CLI

让 Agent 进入真实环境，完成可验证、可交付的动作。

Workflow + Stability

决定它能否从一次成功，变成长期可靠的业务系统。

真正有产品价值的 Agent，不在于“像人一样回答”，而在于“能稳定调动能力去完成任务”。
