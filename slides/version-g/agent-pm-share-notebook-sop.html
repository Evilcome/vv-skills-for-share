02
Chapter 1 / 为什么现在值得关注 Agent

不是更会聊天，而是更能推进任务

对产品经理来说，Agent 真正值得关注的地方不在“更聪明”，而在“能把任务往前推进”，并且把原本依赖多人切换的流程压缩成更清晰的执行链路。

  • 真正可用的 Agent 通常依赖明确目标、外部工具和可执行流程，而不是单次问答。
  • 判断它是否有价值，可以看它是否减少人工切换、减少返工、缩短交付链路。
  • 这件事对 PM 友好，是因为它本质上仍然是“任务拆解 + 协作设计”。
PM 视角的翻译
Agent 更像一个会调用能力、按 SOP 协作的数字同事。

为什么 PM 适合理解

你本来就在做任务拆解、角色分工、流程设计，这与 Agent 的落地方式高度相似。

03
Skill First
先把一类任务稳定地解决好，比先追求万能 Agent 更实际。
定义

什么是 Skill

针对某类任务的标准做法，不只是 prompt，而是一整套边界、输入、判断、输出与异常处理。

价值

先做高频场景

高频、边界清晰、结果可验证的问题，更适合先做成 Skill。

方法

先单点后串联

先把单点任务跑稳，再考虑是否需要 Workflow 串更长流程。

对 PM 的意义

先做最小闭环

这和产品迭代一样：先验证最小闭环，再逐步扩展复杂能力。

04
Chapter 2 / Skill 不只有一种做法

平台、CLI、桌面端都能做 Skill

Platform

平台内配置

适合快速试用、低门槛上手，用图形化方式先验证“这个能力是否有价值”。

CLI

CLI 方式组织

适合把输入、输出、工具调用与流程控制做得更明确，也更容易接进工作流。

Desktop

桌面端工具

例如 Mira 这类形态，适合让非研发角色也能在熟悉界面里制作与运行 Skill。

关键理解

对 PM 来说，关键不是记住某个产品，而是理解：Skill 是可封装、可复用、可迁移的能力单元。

安装与复用

先学会 `skill list`，知道有哪些现成能力，比一开始从零写更高效。本次分享里也会直接用到一个 Skill。

Chapter 3 / 基础设施认知

feishu-cli 为什么能降低 Agent 落地门槛

  • 它的价值不是“操作飞书”本身，而是把文档、评审、写入等动作纳入可执行流程。
  • 它降低了把中间产物接入 Workflow 的门槛，让很多原本要手动搬运的环节变得可串联。
  • 对 PM 来说，它让“文档协作”第一次真正变成 Agent 流程的一部分。
1

读文档

把已有内容接进流程，减少复制粘贴的人肉搬运。

2

写文档

把大纲、结果、记录同步回飞书，形成可 review 的中间产物。

3

接评论

把评审意见变成真实输入，而不是口头反馈与记忆传递。

为什么建议 PM 掌握一点 Git 和 CLI
基础能力 带来的变化 对 Workflow 的帮助
看懂命令 理解输入输出，不再只停留在“帮我试试”。 更容易参与 Skill 调试与流程设计。
会跑基础 CLI 能把想法快速验证成一次真实执行。 缩短“想到流程”和“跑通一次”之间的距离。
了解 Git 知道产物如何被记录、比较、回滚与协作。 更容易理解中间产物 review 的价值。
不是让 PM 变工程师
而是让 PM 更容易进入“能亲手跑通一次”的工作状态。

轻量掌握就够了

会看、会跑、会理解输入输出，已经足够大幅降低你使用 Agent 和设计 Workflow 的门槛。

Chapter 4 / 案例一：客服问答 Skill

简单问题不必过度拆解，关键是先做成稳定能力

客服问答这类问题通常比较简单，不需要一上来设计复杂链路。它更像 FAQ 产品化：先把高频问题回答稳定、边界管清楚。

  • 目标不是“像人一样自由发挥”，而是尽量稳定地给出正确答复。
  • 问题域集中、重复率高、判断逻辑相对明确，天然适合 Skill 化。
  • 先做出稳定命中的能力，比先堆架构复杂度更重要。

修正逻辑

Skill 出问题时，关键是让 LLM 在上下文里修正

  • 优化 Skill 不只是“人改 prompt”，也可以让 LLM 来辅助修正 Skill。
  • 真正关键的是在出问题的上下文里改，而不是脱离现场做抽象优化。
  • 这样更容易定位问题来源，也更贴近真实迭代方式。
A

拿到错误样本

先看问题出在哪个上下文。

B

定位缺失约束

判断是边界、知识还是输出规则缺失。

C

让 LLM 辅助修正

在真实上下文中补规则、改策略。

D

回放验证

看修正是否真的提升命中率。

Chapter 5 / 复杂任务如何做对

前端 Slide 制作，是一个典型复杂任务

  • Slide 不是单一输出，而是同时涉及内容结构、页面表达、视觉风格和可交付质量。
  • 如果一开始就让 Agent 直接生成整套成品，很容易偏题、返工、难评审。
  • 更好的方式是分阶段收敛：先解决结构，再解决视觉，再解决交付。
S

Structure

章节是否完整、叙事是否成故事。

V

Visual

风格是否稳定、信息层级是否清晰。

H

HTML

页面是否可展示、可运行、可交付。

Chapter 6 / Workflow 的真实价值
Workflow 不只是把 Skill 串起来，而是把 SOP 写成可执行流程。
理解 1

串 Skill 只是路径

Workflow 把 Skill 串联起来，是自动化流程的一种实现方式，但不是唯一目的。

理解 2

SOP 更重要

灵活的前提是 SOP 本身写得好，否则只是把混乱流程自动化。

理解 3

旧式固定编排

像 coze、dify 这类更偏固定节点编排，灵活性有限。

理解 4

新式流程组织

现在更接近“写 SOP”，流程可以更灵活，但更考验设计质量。

Chapter 8 / 最后带走什么

一页方法总结：产品经理如何开始做 Agent 落地

1

选场景

先找高频、边界清晰、结果可验证的问题。

2

做 Skill

先把单点能力跑通，不要一开始做大而全。

3

写 SOP / 串 Workflow

复杂任务才需要流程化，把中间产物和交接点设计进去。

4

设评审闭环

把确认前置，在真实上下文中持续修正。

一句话结论

好的 Agent 方案，不是最复杂的那个，而是最容易稳定交付结果的那个。

PM 应该学会的

重点不是记住技术名词，而是学会如何拆任务、设节点、控返工，把复杂问题变成可执行的小步骤。
