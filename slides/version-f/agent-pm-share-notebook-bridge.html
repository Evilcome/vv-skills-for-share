核心命题

02
Chapter 1 / 先回答一个产品问题

为什么很多人一提 Agent，就先想到复杂链路

常见直觉是：要做 Agent，就得规划、多轮决策、调用很多工具、串很长链路。于是 Agent 被理解成高门槛工程系统，而不是可拆解的能力组合。

  • 讨论容易停在概念层，听起来先进，但落不到具体业务动作。
  • 产品经理会天然把它视作“之后再说”的复杂能力，而不是现在就能开始验证的方法。
  • 一旦默认复杂架构先行，就很难从高频小问题里积累可复用经验。
常见误区
把 Agent 理解成“大而全系统”，会直接压缩产品试错空间。

PM 的风险

当讨论先被架构体量绑住，团队往往在真正进入验证前，就先把机会成本抬高了。

03
Core View
很多真实问题，不需要先追求复杂 Agent 架构。
判断 1

先拆 Skill

更有效的起点往往是：把问题拆成边界清晰、职责明确的可复用能力。

判断 2

再组织流程

当任务跨越多个阶段时，再用 Workflow 串联、校验、协作，而不是一把生成到底。

判断 3

重点看可验

对 PM 来说，关键不是懂完所有技术细节，而是会判断问题怎么拆、流程怎么控、结果怎么验。

落地价值

从可验证开始

先做出稳定命中高频问题的能力单元，再逐步扩展复杂度，是更现实的产品路径。

04
Chapter 2 / 用产品经理熟悉的语言理解 Agent

把 Agent 看成“角色 + 工具 + 交付标准”

  • Skill 可以理解为一个职责定义清晰、可复用的角色能力。
  • Tool 是这个角色调用的外部能力，比如读文件、生成页面、写文档。
  • 稳定性 不是只看模型强不强，而是输入边界、工具约束、输出标准是否清晰。
R

Role

知道自己该解决什么问题，不抢别的任务。

T

Tool

能调用哪些外部能力，边界有多大，失败怎么处理。

D

Definition

交付物长什么样、算不算通过、能不能交接给下一环。

05
Workflow / 不是技术炫技

Workflow 更像产品流程设计，而不只是技术编排

1

Planning

先对齐目标、范围、成功标准，避免还没理解任务就直接生成。

2

Task List

把复杂任务拆成几个可执行单元，每一步都能被看见和调整。

3

Review

在关键节点确认方向，让结构问题尽早暴露，而不是尾部返工。

4

Release Check

上线前检查结果完整性与可交付性，让产物真正进入业务流程。

目标与范围先统一问题定义
阶段产物中间件就是控制点
交付形式谁接、怎么用
协作介入在适合阶段插人
返工控制问题定位更清楚
06
PM 最该关注的三个判断标准
判断点 应该追问什么 如果忽略会怎样
值不值得拆成 Skill 这是高频、边界清晰、可复用的问题吗？ 容易把多类任务塞进一个大 Prompt，后续难维护。
何时插入中间确认 哪个阶段最容易方向错、返工贵、多人要协作？ 一次生成失败后，很难判断问题出在哪一层。
产物能否被评估与交接 结果是否有结构化标准，能否被 review、复用、迭代？ 只看 demo 成功，无法沉淀为长期能力。

一句话翻译

PM 不必先学最复杂的 Agent 术语，但一定要会判断：问题拆得对不对，流程卡点放得对不对。

判断顺序建议
先价值 再边界 再流程 最后扩复杂度
07
Chapter 3 / 案例一：客服问答 Skill

客服问答为什么适合先从 Skill 做起来

客服问答的核心目标不是“会推理”，而是“答得准、答得稳、覆盖常见问题”。知识边界通常比较清晰，流程也相对固定。

  • 这类问题更像稳定执行，而不是开放探索。
  • 适合先抽成专用 Skill，而不是上来做通用复杂 Agent。
  • 只要常见问题能持续命中，业务价值就已经很高。
目标
答得准

比“回答花哨”更重要。

边界
边界清

问题域可枚举、可维护。

流程
流程稳

命中、回复、兜底都易定义。

价值
见效快

适合先做业务验证闭环。

08
实现关键 / 知识承载

reference 文件承载知识，不一定先上向量数据库

  • 如果知识规模可控、结构稳定，reference 文件就足以承载主要知识。
  • 实现成本更低、维护更直接、排查问题更容易。
  • 产品经理可以更快验证：知识组织方式到底是不是有效。
为什么这很重要
方案 启动成本 维护透明度
reference 文件 低 高
向量数据库 中-高 中

对 PM 来说，这意味着可以先验证“知识组织”是否有效，而不是过早投入复杂基础设施。

09
Case 1 / 启发

这个案例的启发：先求可控，再求复杂

命中率
高频问题有没有稳定解决
维护成本
知识能不能被直接更新和追踪
迭代速度
能不能快速修正错误与补充覆盖
第一阶段的目标

Skill 的第一价值是把高频问题稳定解决，而不是展示系统复杂度。简单方案只要足够支撑目标效果，就是更好的第一阶段方案。

PM 的观察点

优先关注命中率、维护成本、迭代速度、异常问题定位效率，而不是先被底层架构名词吸走注意力。

Chapter 4 / 案例二：前端 Slide 制作 Skill

为什么 Slide 制作比客服问答更复杂

  • 它不只是回答问题，而是同时生成结构、内容、视觉和页面实现。
  • 用户对结果的主观要求更高，质量波动也更明显。
  • 这类任务更能体现：Skill 的边界定义和约束设计有多重要。
复杂度来源
S

Structure

章节是否完整、信息是否成故事。

V

Visual

视觉风格是否稳定、层级是否清楚。

H

HTML

页面是否可展示、可运行、可交付。

11
多版本迭代的重点
重点不只是“模型更强”，而是“约束更清楚”。

每次迭代本质上都在解决具体问题：结构散、视觉弱、层级不清、页面不可用。

输入

输入格式更规范，减少模型自己猜任务。

输出

输出结构更明确，方便 review 与继续加工。

原则

设计原则更具体，避免“看起来都差不多”的泛化结果。

失败模式

失败更可控，知道该怎么拆 slide、怎么防止溢出与不可用。

12
PM Role / 如何参与迭代

产品经理可以如何参与这种 Skill 打磨

1

定义可用

不是简单说“好不好看”，而是明确是否满足信息表达目标。

2

抽取负反馈模式

结构散在哪、返工点在哪、哪些约束缺失，是最关键的输入。

3

建立评审标准

结构完整性、内容准确性、视觉一致性、可交付性都要可判断。

4

推动迭代闭环

让反馈回到 Skill 本身，而不是每次都靠人工兜住结果。

13
Chapter 5 / 复杂任务与 Workflow

复杂任务为什么不适合一把生成到底

  • 任务越复杂，目标越容易混杂：内容正确、结构清楚、形式合适、协作顺畅，往往不是同一层要求。
  • 全部揉进一次生成，问题难定位、反馈难吸收、返工成本高。
  • 一旦失败，很难判断是需求不清、Skill 不行，还是流程缺少中间确认。
One-shot 的典型后果
一步生成 分阶段 Workflow
问题定位 难 清楚
吸收反馈 弱 强
返工成本 高 低
14
Workflow 的优化价值

把中间产物变成控制点

比如先产出 outline，再做 slides；先 review 结构，再优化视觉。中间产物本身就是协作界面，也是质量控制节点。

outline review visual pass delivery check
A

先解决讲什么

对齐结构和论点。

B

再解决怎么呈现

控制页面与视觉表达。

C

再做协作修改

让不同角色在正确节点介入。

D

最后交付验收

保证产物可展示、可复用、可交接。

15
用产品经理熟悉的机制理解 Workflow
Workflow 环节 对应 PM 机制 作用
planning 先明确目标和范围 减少任务误解，控制边界。
task list 把复杂任务拆成可执行单元 降低复杂度，方便分工。
review 关键节点确认方向 避免尾部大返工。
pre-release check 发布前检查 把结果变成真正可交付的产物。
核心翻译
Workflow 不是技术炫技，而是把产品协作机制显式化。

所以重点不是链路长

而是每一段有没有明确职责、有没有中间产物、有没有可判断的控制点。

Chapter 6 / 完整收尾案例

这次分享的 Slide，本身就是一个 Workflow 产物

  • 先基于分享目标和受众，生成结构化 outline。
  • 再用 Frontend Slides Skill 把大纲转成可展示页面。
  • 再结合 Feishu 协作进行补充、修改和对齐。
  • 复杂交付物可以由多个清晰能力和阶段共同完成。
实际流程
1

outline

先解决讲什么、怎么讲。

2

slides

再解决页面、内容与视觉实现。

3

review

通过协作补齐问题、校准方向。

4

delivery

得到可展示、可继续修改的成品。

17
为什么更稳
各阶段职责清晰后，质量更容易提升，协作更容易展开。
大纲阶段

先解决“讲什么、怎么讲”的问题。

页面阶段

再解决“怎么呈现得更好”的问题。

协作阶段

解决“是否符合目标、是否需要修改”的问题。

交付结果

每一步都可 review、可调整、可接力，整体风险更低。

18
Closing / takeaway

复杂问题不一定靠更长链路解决，而是靠更好的 Skill 拆分与 Workflow 设计

对产品经理来说，理解 Agent 落地的关键，不是先学会最复杂的架构，而是学会判断：问题怎么拆、能力怎么封装、流程怎么协作。

从高频任务开始 先做可复用 Skill 复杂了再加 Workflow
下一步建议
  • 从一个高频、边界清晰的业务任务开始。
  • 先做出可复用 Skill，跑通命中、维护、迭代闭环。
  • 当任务复杂度上升，再增加中间确认与协作节点。
