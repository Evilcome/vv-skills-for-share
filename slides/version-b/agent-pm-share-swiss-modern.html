Agent, Skill, Workflow

今天不从技术实现讲起，而是从产品视角看：为什么 agent 是新交互范式，skill 是高频落地单元，workflow 则让能力变成稳定的业务生产线。

For PM
不是一个更聪明的按钮，而是一套可以被组织的执行系统。
Agent

能理解目标、拆解任务、选择动作。

Skill

把高频问题压缩成可复用能力。

Workflow

把多个动作串成稳定闭环。

Why now

交互入口从页面跳到目标本身。

03
Agent / definition

Agent 不是聊天框升级版

如果说传统 AI 功能是在一个页面里回答问题，那么 agent 更像一个能够为了目标主动行动的“执行体”。用户给它的不是步骤，而是结果。

  • 它接收的是目标，例如“帮我做一个分享 slide”。
  • 它会自己拆分子任务，决定先读什么、用什么工具、如何验证。
  • 它可以在执行过程中根据上下文调整路径，而不是死板跑脚本。
  • 它的核心价值是缩短“从意图到结果”的距离。
Classic AI vs Agent
A
传统 AI回答一个问题，输出一段内容。
B
Agent理解目标，调用工具，持续推进直到交付。
C
产品变化入口不再是固定页面，而是围绕任务设计一整个执行体验。
04
Agent / anatomy

从产品角度拆开 Agent

01 Context

先理解用户是谁、目标是什么、什么算完成。

02 Reason

把复杂目标拆成若干中间动作，并动态调整策略。

03 Act

调用工具、读写文件、触发命令、访问外部系统。

04 Verify

检查结果是否可用，再决定结束还是继续迭代。

所以 PM 真正要设计的，不只是一个 prompt，而是 context、tooling、feedback、guardrail 四件事一起工作的产品系统。

05
Skill / role

Skill 是 Agent 的压缩能力包

Agent 很通用，但通用也意味着不稳定。Skill 的作用，是把某类高频任务里的最佳做法预先固化下来，让 agent 在这个问题上更像专家，而不是临场发挥。

  • 定义什么时候触发、什么问题适合用它。
  • 规定流程：先看什么，再做什么，最后怎么验收。
  • 把领域知识、边界条件、坑点都提前写清楚。
  • 降低 agent 在复杂任务中的探索成本和随机性。
一句话理解
Skill = 为一个重复问题准备好的标准作业指导书。

对于产品经理来说，它很像把“专家经验”产品化：用户未必知道方法，但系统知道该怎么走。

06
Skill / demo

Skill Demo: 读飞书文档

如果没有 skill，agent 遇到“帮我看看这个飞书文档”时，可能需要自己猜 URL、猜 API、猜格式。加上 skill 后，系统直接知道这是一个“只读文档分析任务”。

1
识别意图用户说“查看 / read / 分析 飞书文档”。
2
加载技能明确支持 docx、sheet，只读、不修改。
3
标准输出转成 markdown，再分析结构、提炼重点。
skill trigger 
# 当用户提到这些词，优先命中 skill
keywords: 查看 / 阅读 / read / 飞书文档 / sheet
mode: readonly
supports: docx, sheet, knowledge base
workflow: parse -> markdown -> analyze
output: structure + key points
07
Skill / why enough

大部分情况下，Skill 就够用了

高频

用户问题高度重复，例如读文档、生成 slide、改代码、查表格。

窄域

问题边界清晰，输入输出相对固定，不需要跨很多系统联动。

可控

通过规则、提示词和工具约束，就能把成功率拉到足够高。

  • 从产品 ROI 看，skill 的性价比通常最高：开发快、见效快、可复用。
  • 先把 20 个高频任务做成 skill，往往比直接做“万能 agent”更容易跑通价值。
  • 只有当任务开始跨角色、跨工具、跨时间链路时，workflow 的必要性才会真正出现。
08
CLI / role

CLI 让 Agent 真正能做事

对 PM 来说，CLI 可以理解成一套“低摩擦动作接口”。它让 agent 不必永远停留在语言层，而是可以真正进入工作现场：看文件、改文件、跑命令、调用已有脚本。

  • 相比 GUI，CLI 的动作更明确、可组合、也更容易被自动化。
  • 对内部效率场景尤其重要，因为大量资产本来就存在代码库、脚本、配置中。
  • CLI 不是给用户看的，而是给 agent 一双可执行的手。
PM 可以这样理解 CLI
A
没有 CLIAI 只能“建议你怎么做”。
B
有 CLIAI 可以“直接帮你做完，然后把结果给你”。
C
产品价值交付感更强，用户对结果的感知显著提升。
09
CLI / demo

CLI Demo: cli-server

比如用户说“帮我改一个文案页并打开预览”。对人来说这是一串动作；对 agent 来说，这是一个可连续执行的任务链。

Read

先读目标文件，理解页面结构和已有风格。

Edit

按约束修改文案或布局，尽量保持一致性。

Run

启动预览或执行测试，确认页面没有坏。

Report

返回结果、路径、风险点和下一步建议。

tool chain 
# 一个典型的 agent 执行链
read(landing.html)
edit(headline, CTA, layout)
run(preview server)
verify(viewport, console, visual fit)
reply(what changed + where + next step)
10
Decision model

什么时候该上 Agent，什么时候先别上

适合做 Agent

目标明确但步骤因人而异，需要系统根据上下文自己决定怎么完成。

适合做 Skill

问题高频重复，边界清晰，已知最佳实践可以被沉淀下来。

适合做 Workflow

任务跨多个步骤/系统/时间点，需要稳定衔接和状态管理。

不该做 AI

如果规则完全确定、分支极少，一个传统功能或脚本就足够了。

11
Workflow / role

Workflow 把一次成功变成稳定成功

单次任务做成，不等于可以稳定交付。Workflow 的价值，是把多个 skill、多个工具、多个判断点串起来，确保系统在更长链路里仍然可控。

  • 它关心的不只是“会不会做”，而是“能不能持续做对”。
  • 它把上下游依赖、状态流转、失败重试、人工介入点都设计进去。
  • 它更像业务流程设计，而不是单点能力设计。
一句话理解
Workflow = 把 agent 能力编排成一条可追踪、可复跑、可治理的链路。
12
Workflow / demo

Workflow Demo: 从需求到交付

想象一个典型内部流程：收到需求后，要整理资料、生成方案、产出页面、检查质量、最后回传结果。单个步骤都不难，但一串起来就容易不稳定。

1
输入收集读 PRD、看文档、拉取上下文。
2
技能调用命中合适的 skill，生成中间产物。
3
自动验证跑检查、看错误、评估是否达标。
4
结果回传输出结果，并给人工决定是否继续。
为什么要 Workflow

因为用户真正感知到的不是某个 skill 很聪明，而是整个任务链条是否顺畅、是否少返工、是否值得信任。

Workflow 让“聪明”变成“稳定交付”。

13
Stability / challenge

真正难的不是能力，而是稳定性

不确定输入

用户不会按你期待的格式提需求，信息经常缺失、含糊、矛盾。

不确定过程

模型有随机性，工具有失败率，外部环境也会变化。

不确定标准

“可用”很多时候不是二元判断，而是一个需要被定义的主观阈值。

所以稳定性设计，本质是把不确定性收敛到产品可承受范围内，而不是幻想它完全消失。

14
Stability / design

稳定性设计的 4 个抓手

1
边界清楚定义什么时候能做、什么时候不能做。
2
分层先 skill，后 workflow，再决定是否需要更强 agent。
3
验证每个关键步骤都要有检查点，而不是只看最终输出。
4
兜底失败时能重试、降级、求助人工，而不是静默出错。
guardrails 
# 一个简单但实用的稳定性思路
if input_missing:
  ask_or_infer_defaults()
elif high_risk_action:
  require_confirmation()
else:
  execute()

verify_each_step()
retry_or_fallback()
return_clear_status()
15
Stability / demo

稳定性 Demo: 不是“最聪明”，而是“最稳妥”

一个好 agent 不会每次都秀操作，而是知道什么时候直接做、什么时候先查上下文、什么时候只问一个关键问题、什么时候干脆拒绝。

  • 默认先读代码、读文档，再决定方案，而不是马上下结论。
  • 有风险的操作不擅自做，例如生产、账单、安全类修改。
  • 对模糊需求尽量先做不受阻部分，只在真正卡住时才提问。
  • 最终输出不只给答案，还会说清楚改了什么、为什么这么改。
对 PM 的启发
好的 agent 体验，核心不是 wow moment，而是 trust moment。

用户愿不愿意继续交任务，取决于它是否稳定、透明、可预期。

16
Product lens

产品经理可以怎么参与这件事

定义任务单元

识别哪些用户任务是高频、高价值、可标准化的 skill 候选。

设计反馈机制

让用户知道系统现在在做什么、为什么这么做、做到哪一步了。

设定验收标准

把“好不好”从抽象感觉变成清晰、可判断的结果定义。

补上治理能力

包括权限、风控、失败处理、人工接管和数据闭环。

17
Takeaway

最后用 3 句话总结

1
Agent让交互从“告诉系统怎么做”变成“告诉系统要什么结果”。
2
Skill是高频任务最实用的落地方式，先把专家经验做成产品能力。
3
Workflow决定这套能力能否从 demo 走向稳定业务价值。
一个判断顺序

先找高频任务做 skill，再把跨步骤任务做 workflow，最后在必要场景上做更通用的 agent。

别一开始就追求“全能”，先追求“稳定且有价值”。
