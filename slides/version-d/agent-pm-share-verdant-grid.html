1 / Agent

什么是 Agent

Agent 的关键，不是“会回答”，而是“会推进”。用户输入的是目标，系统自己判断下一步该做什么，直到交付一个可以使用的结果。

  • 传统 AI 偏向一次性输出；Agent 偏向连续执行。
  • 它会结合上下文，主动拆分子任务，而不是等用户手把手下指令。
  • 它依赖工具与反馈回路，不只是一个 prompt 包装层。
  • 产品价值来自“缩短从意图到结果的距离”。
观点
Agent 是围绕任务组织的一种新交互，而不是一个更大的输入框。

所以在产品设计上，要关注目标、动作、状态、反馈和验收，而不只是回答质量。

04
1 / Agent

Agent 怎么工作

Context

先理解用户是谁、目标是什么、约束和成功标准是什么。

Reason

把复杂目标拆成若干步骤，并动态决定最合理的路径。

Act

调用工具、读取资产、执行命令、产出中间结果。

Verify

检查结果可用性，再决定结束、重试还是继续迭代。

PM 真正设计的不是一句 prompt，而是 context、tooling、feedback、guardrail 一起组成的执行系统。

2 / Skill

Skill 的作用

Skill 可以理解成 Agent 的“压缩能力包”。它把高频任务里的最佳做法、触发条件、操作顺序和边界条件，提前写成一份标准作业指导书。

  • 什么时候触发、适合解决什么问题，都在 skill 里提前定义。
  • 把领域知识和易错点前置，减少临场探索成本。
  • 让通用 Agent 在特定问题上更像专家，而不是即兴发挥。
  • 对产品来说，skill 是最适合规模化沉淀经验的单元。
skill = reusable expertise 
# 一个典型 skill 会明确这几件事
trigger: 什么意图触发
scope: 解决什么问题
workflow: 先看什么 / 再做什么 / 怎么验收
guardrail: 边界、禁区、风险点
output: 最后应该给用户什么

3 / Decision

大多数时候，Skill 就够了

高频

问题重复度高，比如读文档、生成 slide、查表、改文案、做结构化分析。

窄域

输入输出相对稳定，边界清晰，不需要跨很多系统或长链路协作。

可控

只要加上规则、模板和工具约束，就能把成功率拉到足够高。

  • 从 ROI 看，skill 往往是最先值得做的：开发快、见效快、复用性强。
  • 先做 20 个高频 skill，通常比直接追求“万能 Agent”更快跑出价值。
  • 只有当任务开始跨角色、跨系统、跨时间节点时，workflow 的必要性才会上升。
08
4 / CLI

CLI 的作用

CLI 可以把它理解成 Agent 的动作接口层。没有 CLI，Agent 只能“告诉你怎么做”；有了 CLI，Agent 才能“直接替你做完”。

  • 读文件、改文件、跑脚本、执行命令，都是可组合、可自动化的动作。
  • 对于内部效率场景尤其关键，因为大量资产本来就在代码库和脚本系统里。
  • CLI 的价值不在界面，而在于给 Agent 一双可执行的手。
对比
没有 CLI，AI 停在建议层；有 CLI，AI 才进入交付层。

所以 CLI 不是补充能力，而是让 Agent 从“会说”到“会做”的关键桥梁。

5 / Workflow

Workflow 自动化

单点能力做成，不等于可以稳定交付。Workflow 的作用，是把多个 skill、多类工具和多个判断点组织成一条可以持续复跑的链路。

  • Workflow 关注的不只是“会不会做”，而是“能不能持续做对”。
  • 它需要处理状态流转、失败重试、人工介入点和上下游依赖。
  • 它更像业务流程设计，而不是单个能力设计。
一句话理解
Workflow = 把 Agent 能力编排成一条可追踪、可治理、可复跑的业务链路。

5 / Workflow

Workflow 链路

1
输入收集读 PRD、看文档、拉历史上下文，先把任务补齐。
2
能力路由命中合适的 skill 或工具，产出中间结果。
3
自动验证跑检查、看错误、评估结果是否达标。
4
人工接管如果风险高、结果模糊或冲突，就把决策权交还给人。

6 / Stability

稳定性设计

Agent 产品真正难的，不是偶尔做对，而是大部分时候都做得足够对。稳定性设计，本质上是在管理不确定性。

  • 输入不确定：用户表达常常缺信息、含糊、矛盾。
  • 过程不确定：模型有随机性，工具会失败，环境会变化。
  • 标准不确定：很多结果不是简单的对错，而是主观阈值。
原则
不是消灭不确定性，而是把不确定性收敛到产品可接受范围内。

因此稳定性不是补充项，而是 Agent 体验的底盘。

14
6 / Stability

稳定性抓手

1
边界清楚定义什么时候能做、什么时候不能做。
2
分层先 skill，后 workflow，再决定是否需要更通用 Agent。
3
验证每个关键步骤都有检查点，而不是只看最终输出。
4
兜底失败时可重试、可降级、可求助人工，而不是静默出错。
stability mindset 
# 一套简单但实用的 guardrail 逻辑
if input_missing:
  ask_or_infer_defaults()
elif high_risk_action:
  require_confirmation()
else:
  execute_safe_steps()

verify_each_step()

16
Takeaway

三个判断

1
Agent让交互从“告诉系统怎么做”变成“告诉系统要什么结果”。
2
Skill是高频任务最实用的落地方式，先把经验做成产品能力。
3
Workflow + 稳定性决定这套能力能不能从 demo 走向真实业务价值。
