Chapter 1

Agent 的价值不在更聪明，而在更能推进任务

  • 对产品经理来说，真正有价值的 Agent 不是“会聊天”，而是“能把事往前推”。
  • 它通常依赖明确目标、外部工具和可执行流程，而不是一次性问答。
  • 判断一个 Agent 是否有价值，可以看它是否减少人工切换、返工和交付耗时。
Classic AI vs Agent
A
聊天型 AI更多是在一个点上给回答。
B
可执行 Agent会为了结果主动选择动作和路径。
C
产品意义交互从页面能力变成任务完成能力。
04
Chapter 1

产品经理天然适合理解 Agent

  • PM 本来就在做任务拆解、角色分工、流程设计，这和 Agent 的工作方式高度相似。
  • 可以把 Agent 理解成“一个会调用能力、按 SOP 协作的数字同事”。
  • Skill 像标准化能力卡，Workflow 像跨角色协作流程，工具像外部系统接口。
一个直观映射
需求流转之于 PM，正如执行链路之于 Agent。

理解这层映射后，很多抽象概念会立刻变得具体：不是学新宗教，而是学一种新的组织方式。

05
Chapter 2

先从最小单元理解：什么是 Skill

标准做法

Skill 是“针对某类任务的稳定解法”，不是泛泛的提示词堆砌。

结构完整

一个好 Skill 往往包含边界、输入要求、判断逻辑、输出格式和异常处理。

先做高频

比起追求一个万能 Agent，更现实的方式是先把高频场景做成可靠 Skill。

可以把它理解为：把“某件事怎么稳定做好”写成一张随时可调用的能力卡。

06
Chapter 2

Skill 不止一种做法，关键是可封装可复用

  • 做 Skill 有很多路径，不一定非要依赖某一个平台。
  • 可以在平台内配置，也可以通过 CLI 组织，或在桌面端工具中完成。
  • 对 PM 最重要的认知不是记住某个产品，而是理解 Skill 是可迁移的能力单元。
三种常见载体
1
平台适合快速配置、轻量试验。
2
CLI适合更接近真实工作现场的执行。
3
桌面端适合更贴近日常办公形态的体验封装。
07
Chapter 2

先学会复用，往往比从零开始更重要

  • Skill 不只是自己做，也可以通过已有列表去发现、安装和复用。
  • 对初学者来说，先掌握 `skill list` 这类入口，比一开始自己写更高效。
  • 更重要的是建立“发现 -> 调用 -> 迭代”的习惯，而不是追求第一次就完美。
推荐心智
先看现成能力，再决定要不要自建。

这和产品工作很像：先看有没有成熟方案，再决定是否值得定制开发。

08
Chapter 3

feishu-cli 为什么能降低门槛

  • 它的价值不只是“操作飞书”，而是把文档、评审、写入纳入可执行流程。
  • 很多原本要手动搬运的中间产物，可以被串进 Workflow 里持续使用。
  • 对 PM 来说，文档第一次真正成为 Agent 流程中的一环，而不只是结果容器。
连接关系
A
Agent / Skill负责理解任务和决定动作。
B
feishu-cli把文档世界变成可调用能力。
C
共享文档成为可评审、可沉淀、可回看的中间产物。
09
Chapter 3

为什么 PM 值得懂一点 Git 和 CLI

  • 不是要求 PM 变成工程师，而是掌握基础后，很多 Agent 使用门槛会明显下降。
  • 会看命令、会理解输入输出，就更容易参与 Skill 调试和 Workflow 设计。
  • 这类基础能力能缩短“想到一个流程”和“亲手跑通一次”之间的距离。
一句话理解
懂一点 CLI，不是为了写代码，而是为了更直接地和 Agent 协作。
10
Chapter 4

简单问题更适合先做成 Skill

  • 像客服问答、文档读取这类问题，通常并不需要过度设计复杂链路。
  • 关键不是拆得多复杂，而是把高频问题回答稳定、边界讲清楚。
  • 对 PM 来说，这很像把 FAQ 产品化：先把重复问题做成可复用能力。
场景判断
问题越高频、越窄域、越可验证，越适合先做 Skill。
11
Chapter 4

Skill 出问题时，要在真实上下文里修正

  • Skill 的优化不只是“人改 prompt”，也可以让 LLM 帮助定位和修正问题。
  • 真正关键的是在问题发生的上下文里修改，而不是脱离现场抽象优化。
  • 这种方式更容易定位错误来源，也更贴近真实迭代方式。
闭环方式
1
记录问题保留用户提问与错误输出。
2
就地修正基于上下文补规则、补边界、补示例。
3
再次验证确认同类问题是否真的被解决。
12
Chapter 5

复杂任务的关键，不是一口气做完，而是分阶段收敛

Content

先明确结构和叙事，避免一开始就陷入页面细节。

Review

把中间产物显性化，让人能及时评审和纠偏。

Design

在方向确认后再做视觉与版式实现，减少返工。

Deliver

最后再验证交付质量，而不是让结果一次成型。

Slide 生产是典型例子：同时涉及内容结构、页面表达、视觉风格和可交付质量，因此更需要前置确认节点。

13
Chapter 6

Workflow 不只是串 Skill，而是把 SOP 写成可执行流程

  • Workflow 是自动化流程的一种实现路径，但核心从来不是节点越多越好。
  • 它真正解决的问题，是如何把 Skill、工具和评审节点按顺序组织起来。
  • 前提是 SOP 本身写得足够好，否则只是把混乱流程自动化。
关键认知
Workflow 的价值，不是复杂，而是让复杂任务变得可追踪、可治理。
14
Chapter 7

一个长链路 Demo 里，最不能省的是中间产物和评审点

  • 长链路里最怕的不是步骤多，而是没有可回看的确认点。
  • 像飞书链接这样的共享产物，本质上就是 plan / review 节点，便于协作与追踪。
  • 这也是“轻量落地”的关键：先把关键节点串起来，不必一开始就全自动。
典型链路
1
生成提纲先有可讨论的结构雏形。
2
写入飞书形成共享中间产物供评审。
3
根据反馈修正把问题尽量在中段收敛。
4
最终生成把确认过的内容变成交付物。
15
Chapter 8

给 PM 带走的一页方法论

选场景

先找高频、边界清晰、结果可验证的问题。

做 Skill

先把最小能力单元跑通，再考虑是否值得更复杂编排。

写 SOP

复杂任务先写清楚流程，再决定如何串成 Workflow。

设评审

把中间产物和 review 节点设计进去，在真实上下文持续修正。

  • 优先优化的是减少返工和提升协作效率，而不是盲目追求链路复杂度。
  • 先稳住一个小场景，往往比直接追求“大而全”更容易建立组织信心。

Takeaway

先把 Skill 做稳，再让 Workflow 放大结果。

如果今天只记住一句话：复杂 Agent 的核心，不是一次生成成品，而是在真实上下文里逐步做对。
