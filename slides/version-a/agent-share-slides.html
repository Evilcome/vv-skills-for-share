Chapter 1

什么是 Agent

如果只把它看成“会聊天的大模型”，后面的 skill、workflow、稳定性都会无从谈起。

Agent 不是一句话，Agent 是一个运行时。
02
Definition

Agent = 模型 + 上下文
+ 工具 + 状态 + 策略

  • 模型负责推理与生成，但不是全部能力来源。
  • 上下文决定它“知道什么”、任务边界在哪。
  • 工具让它不只回答，还能读文件、跑命令、调服务。
  • 状态让多轮执行保持连续，不是每次都从零开始。
  • 策略约束它如何分步、何时停下、何时升级处理。
agent-runtime.ts
// mental model
{
  model: reasoning engine,
  context: task + memory + files,
  tools: read / search / execute,
  policy: plan / verify / recover
}
03
Contrast

为什么 agent 不是
“高级一点的 prompt”

只有 prompt
一次性输入，一次性输出；成功依赖写 prompt 的人；很难复用，也很难追责。
有 agent
能分步、能观察中间状态、能用工具、能根据结果回退或继续。
用户视角
像是在给一个会执行任务的搭档分配工作，而不是在写一次性咒语。
系统视角
像在设计一个有限能力、可被约束、可被验证的执行流程。
如果产品要落地，就不能只讨论“提示词技巧”，而要讨论：能力边界、工具接入、失败恢复和结果验收。
Chapter 2

Skill 的作用

这往往是把 agent 从“聪明但飘”拉回到“可靠且可用”的第一步。

Skill 把经验变成默认路径。
04
Skill

Skill 本质上是在教 agent
“这类题该怎么做”

  • 给任务一个更窄、更明确的入口，减少 agent 到处试。
  • 把成功样式、工具顺序、边界条件预先写进系统。
  • 让不同用户获得更接近的执行质量，而不是全靠运气。
  • 随着使用积累，可以持续优化 skill，而不是重复教育每个用户。
Without Skill vs With Skill
1
Without先猜、再搜、再试错，路径长且不稳定。
2
With直接命中一套成熟流程，工具和输出格式都更可预期。
3
Result更省 token、更省时间，也更适合团队规模化使用。
05
Demo / Skill

一个 skill demo
应该让人看到什么

输入不需要非常精确。用户只要说需求，skill 会自动补足执行路径。
中间步骤是可理解的。不是黑盒神谕，而是可以看到搜索、读取、编辑、验证。
输出风格是统一的。因为 skill 对结构、工具和边界都有约束。
demo transcript
user: 帮我看一下飞书文档里的需求重点
agent: 命中 feishu-cli-read
agent: 读取文档 → 提炼结构 → 输出结论
effect: 用户不需要知道每一步，只需要得到稳定结果
06
Pragmatism

大部分情况下
skill 就已经够用了

很多问题并不需要复杂的多 agent 编排。只要把高频任务的默认路径设计好，已经能覆盖大量真实需求。

read docs analyze table review code generate slides summarize logs
Decision Rule
  • 任务路径相对固定，skill 最适合。
  • 主要价值在于稳定产出，而不是复杂协调。
  • 单次执行就能闭环，不需要长期状态管理。
  • 先把 skill 做好，再决定要不要升级到 workflow。
Chapter 3

CLI 的作用

当 agent 进入真实工作流，它就需要一个“执行界面”和“操作通道”。CLI 往往是最自然的起点。

CLI 让 agent 从对话框走进生产力工具箱。
07
CLI

CLI 不只是输入输出
而是 agent 的可操作外壳

  • 它让工具调用天然成立：文件、终端、git、脚本都在同一个工作面里。
  • 它让过程更透明：每一步读了什么、改了什么、跑了什么都可追踪。
  • 它让人机协作更顺畅：用户随时能中断、确认、修正方向。
  • 它也更适合服务化：同样的动作未来可以挂到 server 或 workflow 上。
cli session
$ ask "做一个 agent 分享 slides"
> detect skill: frontend-slides
> ask structured questions
> generate previews
> build final deck
> open in browser
08
Demo / Server

从 CLI 到 server：
把能力做成可被系统调用的接口

CLI Demo一个人发起，一个 agent 执行，强调交互和即时反馈。
Server Demo把同样能力暴露给别的系统，让流程自动触发，而不是靠人手点。
核心变化交互式工具变成服务式工具，要求更明确的输入、输出和错误处理。
产品价值能力不再只属于“会用 CLI 的人”，而可以进入更广泛的业务流程。
Chapter 4

Workflow 自动化

当你不满足于“今天成功一次”，而是希望“以后反复成功”，workflow 就出现了。

Workflow 把 agent 的单点能力变成系统能力。
09
Workflow

什么时候要从 skill
升级到 workflow

  • 一个任务需要多个阶段，而且阶段之间有依赖关系。
  • 不止一个 agent 或服务参与，必须明确谁先做、谁验收、谁兜底。
  • 需要可观测和可追踪，不能只看最终回答。
  • 需要把失败场景纳入设计，而不是出错后人工救火。
Upgrade Signal
A
单步任务skill 已足够，追求高效和一致性。
B
多步任务开始需要状态转移、阶段校验和失败回滚。
C
系统任务进入 workflow 范畴，目标是自动化和稳定运营。
10
Demo / Workflow

一个 workflow demo
最有说服力的地方

1
Trigger一个需求、一个事件、一次提交，成为流程起点。
2
Dispatch由路由逻辑决定命中哪类 skill、哪种工具、是否需要 server 能力。
3
Verify中间产物被检查，不符合标准就回退或重试，而不是直接往下走。
4
Deliver结果自动落到正确位置，比如 PR、文档、报告或线上服务。
Chapter 5

稳定性设计

越自动化，越不能把成功寄托在模型临场发挥。稳定性不是锦上添花，而是前提。

Reliable agent = constrained agent + verified agent
11
Stability

稳定性来自约束、验证和恢复机制

约束输入问题分类、结构化参数、明确上下文边界。
约束执行规定工具顺序、限制危险动作、必要时让人确认。
验证输出语法检查、测试、规则校验、格式检查。
失败恢复重试、降级、回滚、转人工，不能让流程静默失败。
可以把稳定性理解为：先尽量让 agent 少犯错；再让系统尽快发现错；最后确保即使错了也不会造成不可接受的后果。
12
Demo / Stability

一个稳定性 demo 最好展示
“失败被控制住了”

Case 1
agent 发现上下文不足，于是先问问题，而不是盲答。
Case 2
agent 试图执行危险操作，被策略层拦住并要求确认。
Case 3
输出不符合 schema，自动修正或退回上一阶段重试。
Case 4
最终失败时，系统能给出清晰原因和下一步，而不是沉默地卡住。
对产品同学来说，这比“模型偶尔很聪明”更重要，因为这决定了能力能否真正进入业务。
13
Collaboration

产品和研发
怎么一起定义 agent

  • 产品定义任务价值、边界和验收标准。
  • 研发定义工具接入、执行路径和失败恢复机制。
  • 双方共同决定哪些场景只做 skill，哪些值得做 workflow。
  • 评价 agent 不能只看 demo，要看成功率、可解释性和维护成本。
A Practical Split
P
Product这件事为什么值得做，用户期望看到什么结果。
R
R&D系统如何可靠地把这个结果做出来。
T
Together把成功案例抽象成 skill，把高价值链路抽象成 workflow。
14
Summary

最后用 4 句话
收住今天的主题

1Agent 是一个运行时系统，不只是一个 prompt。
2Skill 是最先该做好的能力层，因为它最直接提升稳定性和复用性。
3Workflow 是把单点能力系统化，用于更自动化、更可观测的任务链路。
4稳定性设计决定了 agent 能不能从 demo 走向真实业务。
